# PageMaker365 Installer Deployment Contract

Status: first implementation contract.

Last updated: 2026-07-08.

## Purpose

This contract defines what the PageMaker365 installer expects from the PageMaker365 control plane before it can deploy a customer-owned runtime environment.

The PageMaker365 control plane remains hosted by PageMaker365. The installer is for customer-owned Azure, Entra, and SharePoint runtime setup.

## Boundary

The installer owns:

- validating the customer deployment package
- signing in to the customer's Azure tenant and Microsoft Graph tenant
- validating Azure subscription, resource group, RBAC, and quota readiness
- validating SharePoint site and document library access
- creating or updating approved Azure runtime resources
- writing customer runtime secrets to customer-owned Key Vault
- returning deployment evidence and smoke-test results

The installer does not own:

- PageMaker365 public website or control-plane hosting
- lead capture, billing, signing, onboarding, or customer portal records
- storing customer runtime secrets in the PageMaker365 control plane
- granting permissions without customer administrator consent
- executing arbitrary AI-generated commands

## Required Deployment Package Sections

The customer install package should be generated by the PageMaker365 control plane after onboarding reaches the deployment export gate.

Required launch sections:

- `contractVersion`
- `customer`
- `azure`
- `sharePoint`
- `app`
- `entra`
- `controlPlane`
- `secrets`
- `features`
- `smokeTests`

The installer currently accepts the older alpha package shape, but preflight warns when launch contract fields are missing. Explicit package hash mismatches and raw secret containers are blocking failures.

## Package Export Trust Metadata

Production packages should include control-plane export metadata so the installer can prove which onboarding export it is using and whether the local file has been edited.

Recommended `controlPlane` fields:

- `deploymentExportId`: immutable export ID from the PageMaker365 control plane.
- `exportedAt`: UTC timestamp when the package was generated.
- `expiresAt`: optional UTC expiration timestamp for time-limited exports.
- `issuer`: PageMaker365 issuer name.
- `issuerEnvironment`: `prod`, `staging`, `sandbox`, or equivalent.
- `onboardingSessionId`: source onboarding session ID.
- `discoveryId`: source tenant discovery payload ID.
- `schemaId`: schema URI used for the export.
- `packageHash`: canonical package hash.
- `packageHashAlgorithm`: currently `SHA-256`.
- `canonicalization`: currently `json-c14n-v1`.
- `publicKeyId`: signing key identifier.
- `signature`: detached signature metadata placeholder.
- `signatureAlgorithm`: signature algorithm metadata placeholder.
- `trustMode`: `UnsignedAllowed` during alpha rollout or `SignedRequired` for strict production exports.
- `jwksUrl`: optional public key discovery endpoint.
- `revocationUrl`: optional export revocation endpoint.
- `correlationId`: server trace ID for support.

The first installer trust pass computes `sha256:<hex>` over canonical JSON with object properties sorted by ordinal name. The hash excludes only `controlPlane.packageHash` and `controlPlane.signature`, so package formatting and property order do not affect the result.

Trust behavior:

- Missing trust metadata is a warning while alpha packages remain supported.
- `controlPlane.trustMode = SignedRequired` makes missing hash/signature/key metadata a failure.
- A declared `packageHash` that does not match the package contents is always a failure.
- Signature fields are recorded now; cryptographic signature verification is a later signing slice.

Generated package provenance behavior:

- `controlPlane.onboardingSessionId` must match the active installer bootstrap session ID.
- `customer.tenantId` must match bootstrap `expectedTenantId` when that bootstrap value is known and not a placeholder.
- `azure.tenantId`, when present, must match the same expected tenant ID.
- `controlPlane.discoveryId` must match the active tenant discovery payload when discovery context exists.
- `controlPlane.deploymentExportId` is required for packages downloaded from the PageMaker365 portal.
- Mismatched generated packages fail before the installer loads customer state or unlocks deployment actions.

## Azure Runtime Resource Contract

The first production Bicep template should support these resources in the customer subscription:

| Resource | Purpose |
| --- | --- |
| Resource group | Customer-owned deployment boundary. |
| Log Analytics workspace | Central runtime logs and metrics. |
| Application Insights | API/runtime telemetry. |
| Key Vault | Runtime secrets and generated keys. |
| Storage account | Runtime assets, support bundles, or export staging if needed. |
| App Service plan or Container Apps environment | Runtime API host. |
| API App Service or container app | Runtime API and backend worker surface. |
| Static Web App or frontend App Service | Runtime frontend if deployed separately. |
| Managed identity | Runtime access to Key Vault and Azure resources. |

The Bicep output must include:

- runtime API URL
- runtime portal URL
- Key Vault URI
- managed identity principal ID
- Application Insights connection string reference name
- deployment resource IDs

## Entra And Graph Contract

The final app registration contract must decide:

- create new app registrations or reuse customer-provided registrations
- single app registration or separate portal/API registrations
- delegated scopes versus application permissions
- `Sites.Selected` versus broad SharePoint permissions
- redirect URIs for customer runtime sign-in
- required admin role for consent
- post-consent verification checks

Initial installer checks validate:

- Microsoft Graph sign-in exists
- Graph tenant matches the customer package
- required Graph scopes are present in the current session
- signed-in account appears to have Global Administrator, Cloud Application Administrator, or Application Administrator role

## SharePoint Contract

The package must include:

- SharePoint site URL
- optional Graph site ID
- default document library name
- target permission model

Preferred production model:

- Use `Sites.Selected` where possible.
- Grant the runtime app access only to the selected customer site.
- Validate the configured library exists before deployment proceeds.

## Secrets Contract

Raw secret values must not be stored in the customer deployment package.

The package may contain:

- names of required secrets
- Key Vault secret names
- labels and descriptions for installer prompts
- whether a secret is generated by installer or supplied by customer

The package must not contain:

- client secrets
- API keys
- connection strings
- Stripe secrets
- ACS Email connection strings
- database passwords
- signing private keys
- refresh tokens

The installer should write supplied or generated values directly to the customer Key Vault and redact them from logs and support bundles.

## Control Plane Evidence Contract

After a deployment attempt, the installer should produce a redacted evidence payload containing:

- installation ID
- deployment export ID
- package hash
- package trust status and signing key ID
- installer version
- started/completed timestamps
- signed-in tenant IDs
- Azure subscription ID
- resource group name
- deployment operation ID
- Bicep what-if result summary
- deployment result summary
- smoke-test result summary
- redacted support bundle path

No raw secrets or auth tokens should be included.

## Smoke Test Contract

Minimum post-deployment smoke tests:

- runtime API health endpoint returns healthy
- runtime frontend loads
- configured SharePoint site resolves through Graph
- configured document library resolves through Graph
- runtime can read required Key Vault references through managed identity
- license validation endpoint accepts the signed license activation
- entitlement sync endpoint returns a signed response
- Application Insights receives one telemetry event

## Open Decisions

These decisions are still required before full production deployment:

- final runtime hosting model: App Service, Container Apps, or Static Web Apps plus API App Service
- final app registration shape and Graph permissions
- final `Sites.Selected` consent flow
- runtime database requirement, if any
- customer DNS and custom domain automation level
- production signing certificate and distribution format
- whether AI diagnostics call the PageMaker365 backend or a local approved diagnostic endpoint
